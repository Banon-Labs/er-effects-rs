# Deobfuscated ER RTTI class map (`deobf-rtti-classmap.tsv`)

A harvest of **every MSVC RTTI vtable → class name** in Elden Ring 1.16.1, for later
**Ghidra symbol sync**. 10,202 vtables resolved (4,586 named `CS`/engine classes; the rest are
`std::_Func_impl`/lambda functor vtables).

## Provenance

- Source: the **dearxan-deobfuscated** mapped image `eldenring-deobf.bin` (gitignored — the
  copyrighted binary is never committed; see `deobfuscated-er-image-tooling-2026`). The class
  **names** here are RTTI metadata (FromSoft's own embedded type descriptors), not game code.
- Generated by `scripts/rtti-scan-all.py`: finds MSVC x64 `CompleteObjectLocator`s
  (`u32[O+0x14] == O`, signature 1), reads each `TypeDescriptor` mangled name, then maps every
  qword that points to a COL → its vtable base (`[vtable-8]` holds the COL VA).
- Format: `0x<vtable_va>\t<mangled_name>` (one per line), VAs are absolute (image base
  `0x140000000`, so on-disk RVA = VA − base). Regenerate with
  `python3 scripts/rtti-scan-all.py docs/recon/deobf-rtti-classmap.tsv`.

## Ghidra import

The VAs are absolute load addresses for the standard `0x140000000` base. To label a vtable in
Ghidra, demangle the `.?AV<Class>@<NS>@@` name (Ghidra's MS demangler handles these) and apply at
the listed address. A small script can iterate the TSV and `createLabel`/`setComment` each vtable.

## Why this exists (the correction that prompted the harvest)

RTTI here **overturned** a chain of misidentifications that several `bd` memories were built on
(see `rtti-correction-0x14078e1c0-is-commandselectdialog-not-title-confirm-2026`):

| Address | Was labeled | RTTI truth |
|---|---|---|
| `0x14078e1c0` | "title confirm router" | `CommandSelectDialog::Update` (vtable slot +0x10; shared by CommandSelectDialog / ChrMakeCommandSelectDialog / CSEventListDialog / DetailKeyGuideDialog — all **in-game** dialogs) |
| `0x142b26468` | (runtime, "0xe00-skewed") | `CS::TitleTopDialog` vtable — the **real** on-disk base (no skew); owner+0xe0 |
| `0x142afa070` | "router_this" | `CS::GestureEquipDialog` |
| `0x142aa1fa0` | — | `CS::CommandSelectDialog` |

Consequence: the `[dialog+0xb0c]` cursor / `[dialog+0x1290]` rows "confirm route" offsets are
CommandSelectDialog-family layout and do **not** apply to the TitleTopDialog — which is why the
title menu's headless drive kept failing. Always validate a function/object identity against this
map (or `scripts/rtti-deobf.py <slot_va>`) before trusting an offset.
